# RoadMate — Deployment Guide

## Quick Start (Development)

### Prerequisites
```bash
# macOS
brew install ruby@3.3 postgresql redis
ruby -v        # 3.3.5
postgres --version
redis-cli ping # Should return PONG

# Linux (Ubuntu)
sudo apt-get update
sudo apt-get install ruby-3.3 postgresql postgresql-contrib redis-server
```

### Initial Setup
```bash
cd roadmate
bundle install                         # Install gems
rails db:create db:migrate            # Setup databases
rails db:seed                         # Seed initial data (optional)
./bin/dev                             # Start dev server
```

Visit `http://localhost:3000`

### Running Services
```bash
# Terminal 1: Web + Jobs + Assets
./bin/dev

# If using separate terminals:
# Terminal 1: Web server
rails server

# Terminal 2: Job processor
bundle exec solid_queue:work

# Terminal 3: Tailwind CSS watcher
./bin/rails tailwindcss:watch
```

## Building for Production

### Docker Build Locally

```bash
# Build image
docker build -t roadmate:latest .

# View build output
docker build -t roadmate:latest . --progress=plain

# Inspect image
docker image ls roadmate
```

### Dockerfile Explanation

The multi-stage Dockerfile:

```dockerfile
# Stage 1: Base (dependencies)
FROM ruby:3.3.5-slim AS base
  # Install system packages (curl, postgresql-client, jemalloc)

# Stage 2: Build (compile gems + assets)
FROM base AS build
  # Install build tools (gcc, git, libpq-dev)
  # Copy Gemfile and bundle install
  # Run bootsnap precompilation
  # Precompile assets (CSS, JS)

# Stage 3: Final (runtime)
FROM base
  # Copy built artifacts from build stage
  # Create non-root user (rails:1000)
  # Run as non-root user (security best practice)
  # EXPOSE 80
  # CMD: Start Thruster + Puma
```

**Benefits**:
- Multi-stage keeps final image small (~500MB vs 1.2GB with build tools)
- Non-root user improves security
- All system dependencies in `FROM ruby:...-slim` (minimal base)

## Kamal Deployment

### Prerequisites

```bash
# Install Kamal
gem install kamal

# Or, use bundled version
bundle exec kamal version

# Docker must be installed and running
docker --version
```

### Configuration

**File**: `config/deploy.yml`

Key sections:

```yaml
service: roadmate                    # Container name
image: roadmate                      # Docker image name
registry:
  server: localhost:5555             # Local registry (change for prod)
  username: your-docker-user         # Optional
  password: KAMAL_REGISTRY_PASSWORD  # From .kamal/secrets

servers:
  web:
    - 192.168.0.1                    # Target server IP (UPDATE THIS)

env:
  secret:
    - RAILS_MASTER_KEY               # From config/master.key
  clear:
    SOLID_QUEUE_IN_PUMA: true        # Run jobs in Puma process

volumes:
  - "roadmate_storage:/rails/storage" # Persist uploads + DB backups

builder:
  arch: amd64                        # Match target architecture
```

### Setup Secrets

```bash
# Create .kamal/secrets (NOT in git)
mkdir -p .kamal
touch .kamal/secrets

# Add secrets (one per line)
# RAILS_MASTER_KEY=<value from config/master.key>
# KAMAL_REGISTRY_PASSWORD=<docker registry auth token>

# Verify
cat .kamal/secrets
```

**Note**: `.kamal/secrets` is in `.gitignore` by default.

### First Deployment

```bash
# 1. Build and push image to registry
bin/kamal build push

# 2. Deploy to server (first time setup)
bin/kamal deploy

# This will:
# - Connect to 192.168.0.1 via SSH
# - Pull Docker image
# - Create container
# - Run migrations
# - Start service
# - Run health checks

# 3. View logs
bin/kamal logs

# 4. SSH into running container
bin/kamal app shell

# 5. Run Rails console
bin/kamal console

# 6. View deployment status
bin/kamal status
```

### Redeploy After Code Changes

```bash
git push origin main

bin/kamal build push       # Build new Docker image
bin/kamal deploy           # Deploy (zero-downtime)

# Kamal will:
# - Pull new image
# - Start new container
# - Wait for health check (/up endpoint)
# - Drain old container (no new requests)
# - Stop old container
# - Keep new container running
```

### Common Kamal Commands

```bash
# Status
bin/kamal status

# Logs (tail)
bin/kamal logs -f

# Database console
bin/kamal app exec --interactive --reuse "bin/rails dbconsole"

# Rails console
bin/kamal app exec --interactive --reuse "bin/rails console"

# Run Rake task
bin/kamal app exec "bin/rake tasks:run"

# SSH access
bin/kamal app shell

# Stop deployment
bin/kamal stop

# Remove all containers
bin/kamal remove

# Rollback to previous release
bin/kamal rollback
```

## Database Setup (Production)

### PostgreSQL Connection

**File**: `config/database.yml`

Production config (auto-managed by Kamal):

```yaml
production:
  primary: &primary_production
    adapter: postgresql
    database: roadmate_production
    username: roadmate
    password: <%= ENV["ROADMATE_DATABASE_PASSWORD"] %>
    host: db.example.com              # External DB server
    port: 5432

  # Separate DBs for cache, queue, cable (same credentials)
  cache:
    <<: *primary_production
    database: roadmate_production_cache
    migrations_paths: db/cache_migrate

  queue:
    <<: *primary_production
    database: roadmate_production_queue
    migrations_paths: db/queue_migrate

  cable:
    <<: *primary_production
    database: roadmate_production_cable
    migrations_paths: db/cable_migrate
```

### Migrations

```bash
# After deploying code with new migrations
bin/kamal app exec "bin/rails db:migrate RAILS_ENV=production"

# Or let Kamal handle it automatically
# (entrypoint script runs migrations)
```

### Backup Database

```bash
# Local backup
pg_dump roadmate_production > backup_$(date +%Y%m%d).sql

# Restore from backup
psql roadmate_production < backup_20260316.sql

# With Kamal (if using external DB)
bin/kamal app exec "pg_dump -h DB_HOST roadmate_production > /rails/storage/backup.sql"
bin/kamal download /rails/storage/backup.sql
```

## Active Storage (File Uploads)

### Development (Local)

**File**: `config/storage.yml`

```yaml
local:
  service: Disk
  root: <%= Rails.root.join("storage") %>
```

**Usage**:
```ruby
class User < ApplicationRecord
  has_one_attached :avatar
end

# In view
<%= image_tag user.avatar, class: "rounded-full" %>

# Upload via form
<%= form.file_field :avatar %>

# Files stored in: storage/
```

### Production (AWS S3 / Cloudflare R2)

**Setup S3 Bucket**:
```bash
# AWS Console
1. Create S3 bucket: roadmate-storage
2. Enable versioning
3. Block public access (keep private)
4. Create IAM user with S3 access
5. Generate access key + secret
```

**Setup R2 (Cloudflare, cheaper than S3)**:
```bash
# Cloudflare Console
1. Go to R2 → Create bucket: roadmate-storage
2. Create API token with R2 access
3. Get account ID, access key, secret key
```

**Configuration**:

```yaml
# config/storage.yml
amazon:
  service: S3
  access_key_id: <%= ENV["AWS_ACCESS_KEY_ID"] %>
  secret_access_key: <%= ENV["AWS_SECRET_ACCESS_KEY"] %>
  region: us-east-1
  bucket: roadmate-storage

cloudflare_r2:
  service: S3
  access_key_id: <%= ENV["R2_ACCESS_KEY_ID"] %>
  secret_access_key: <%= ENV["R2_SECRET_ACCESS_KEY"] %>
  endpoint: <%= ENV["R2_ENDPOINT"] %>  # https://...r2.cloudflarestorage.com
  bucket: roadmate-storage
  force_path_style: true
```

**Environment Variables** (production):
```bash
# AWS S3
AWS_ACCESS_KEY_ID=AKIA...
AWS_SECRET_ACCESS_KEY=...
AWS_REGION=us-east-1
ACTIVE_STORAGE_SERVICE=amazon

# OR Cloudflare R2
R2_ACCESS_KEY_ID=...
R2_SECRET_ACCESS_KEY=...
R2_ENDPOINT=https://account-id.r2.cloudflarestorage.com
ACTIVE_STORAGE_SERVICE=cloudflare_r2
```

**Usage** (same as local):
```ruby
user.avatar.attach(params[:avatar])
user.avatar.url  # Returns S3/R2 URL
```

## SSL/TLS (HTTPS)

### Let's Encrypt (Free, Automatic)

**File**: `config/deploy.yml`

```yaml
proxy:
  ssl: true
  host: roadmate.example.com
```

**Kamal will**:
1. Request SSL cert from Let's Encrypt
2. Renew automatically before expiry
3. Redirect HTTP → HTTPS

### Rails Configuration

**File**: `config/environments/production.rb`

```ruby
config.assume_ssl = true           # Assume HTTPS in URLs
config.force_ssl = true            # Redirect HTTP → HTTPS
config.ssl_options = { hsts: { max_age: 1.year.to_i } }  # HSTS header
```

## Performance Tuning

### Puma Configuration

**File**: `config/puma.rb`

```ruby
max_threads_count = ENV.fetch("RAILS_MAX_THREADS") { 5 }
min_threads_count = ENV.fetch("RAILS_MIN_THREADS") { max_threads_count }
threads min_threads_count, max_threads_count

workers ENV.fetch("WEB_CONCURRENCY") { 2 }  # Multiple processes

# Connection pooling
ActiveRecord::Base.connection_pool_size = max_threads_count
```

**Tuning for 192.168.0.1 (1 server)**:
```bash
# Set in .kamal/secrets or config/deploy.yml
WEB_CONCURRENCY=2           # 2 Puma processes
RAILS_MAX_THREADS=25        # 25 threads per process
SOLID_QUEUE_CONCURRENCY=4   # 4 job threads
```

### Database Connection Pooling

```yaml
# config/database.yml
production:
  max_connections: 25  # Pool size
  checkout_timeout: 2.0
  reaping_frequency: 10
```

### Caching

```ruby
# config/environments/production.rb
config.cache_store = :solid_cache_store, { database: :cache }

# Or with Redis
config.cache_store = :redis_cache_store, { url: ENV["REDIS_URL"] }
```

## Security Checklist

### Production Deployment

- [ ] `RAILS_ENV=production`
- [ ] `SECRET_KEY_BASE` set (in `.kamal/secrets`)
- [ ] `RAILS_MASTER_KEY` set
- [ ] HTTPS enabled (`proxy.ssl: true`)
- [ ] SSL certificate valid (Let's Encrypt)
- [ ] Database password strong (32+ char, random)
- [ ] SSH key authentication (no password SSH)
- [ ] Firewall configured (only 80, 443 exposed)
- [ ] No debug mode in production
- [ ] Logs don't contain secrets
- [ ] Backup strategy in place
- [ ] Health check working (`/up`)

### Database Security

```bash
# Create limited-permission DB user
psql -U postgres
CREATE ROLE roadmate WITH LOGIN PASSWORD 'strong-random-password';
GRANT CREATE ON DATABASE roadmate_production TO roadmate;

# Don't use root/admin for app
```

### Secrets Management

```bash
# Never commit secrets
echo ".kamal/secrets" >> .gitignore

# Store secrets securely
# Option 1: Environment variables (1Password, Vault)
# Option 2: Secret manager (AWS Secrets Manager, Hashicorp Vault)
# Option 3: Manual .kamal/secrets (keep backed up offline)
```

## Troubleshooting

### App Won't Start

```bash
# Check logs
bin/kamal logs

# Check Puma status
docker ps | grep roadmate

# SSH and inspect
bin/kamal app shell
ps aux | grep puma
```

### Database Connection Fails

```bash
# Check DATABASE_URL
bin/kamal app exec "bin/rails dbconsole"

# Test connection
psql -h db.example.com -U roadmate -d roadmate_production
```

### Jobs Not Running

```bash
# Check Solid Queue status
bin/kamal app exec "bin/rails solid_queue:status"

# View job queue
bin/kamal app exec "bin/rails console" << EOF
SolidQueue::Job.limit(10).each { |j| puts j }
EOF

# Retry failed jobs
bin/kamal app exec "bin/rails solid_queue:clear"
```

### Out of Disk Space

```bash
# Check disk usage
bin/kamal app shell
df -h

# Clean up Docker
docker system prune -a

# Remove old releases
docker image prune -a --filter "until=72h"
```

### High Memory Usage

```bash
# Monitor
bin/kamal app shell
top
free -h

# Reduce threads
WEB_CONCURRENCY=1 RAILS_MAX_THREADS=10

# Upgrade server or optimize code
```

## Summary

**RoadMate deployment is simple and cost-effective:**

1. **Development**: `./bin/dev` (Rails server + Solid Queue + Tailwind)
2. **Build**: `docker build` (multi-stage Dockerfile)
3. **Deploy**: `bin/kamal deploy` (zero-downtime, single command)
4. **Monitor**: `bin/kamal logs` + health check (`/up`)
5. **Backup**: PostgreSQL snapshots + Active Storage (S3/R2)
6. **Scale**: Multi-server (when hitting single-server limits)

**For production**, use Kamal with:
- External PostgreSQL (managed service or separate VPS)
- S3/R2 for file storage
- Let's Encrypt for HTTPS
- Basic monitoring (Docker logs + health check)

Estimated first-year cost: **$500-1000** (excluding domain, dev time, SMS costs).

See `/docs/system-architecture.md` for detailed infrastructure diagrams.
